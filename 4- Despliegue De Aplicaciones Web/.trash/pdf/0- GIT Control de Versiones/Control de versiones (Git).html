CONTROL DE VERSIONES

Qu es y para qu sirve?

Git es un sistema de control de versiones libre y distribuido diseado para gestionar proyectos. Su objetivo es controlar y gestionar una enorme cantidad de ficheros de forma fcil y eficiente.

Se basa en repositorios que se inicializan en un directorio concreto y tienen toda la informacin de los cambios realizados.

Partes de un repositorio

El repositorio remoto puede ubicarse en GitHub, GitLab, Bitbucket, SourceForge, entre otros. Es una copia del repositorio de git que se encuentra alojado en un servidor o en un sitio distinto al local.

El repositorio local se divide en:

  • Directorio de trabajo (working directory): donde se almacena el contenido del repositorio.
  • rea de preparacin (staging area o index o cached): donde estn indicados los cambios de confirmacin.
  • Repositorio local (local repository): donde estn todas las versiones de manera local.

Estados de un fichero

  • untracked: Son ficheros que existen en el rea de trabajo pero no existen para Git. No han sido aadidos al seguimiento del repositorio.
  • staged: Son ficheros que se han aadido al staging area y estn listos para ser incluidos en el siguiente commit. Aadidos con git add.
  • committed: Son ficheros que se guardaron en el ltimo commit y que no han sido modificados desde entonces. Estn en el repositorio y no tienen cambios pendientes.
  • modified: Son ficheros que se han modificado desde el ltimo commit pero que an no se han aadido al staging area. Han sido editados en el rea de trabajo, pero Git an no los ha preparado para el commit.
  • ignored: Son ficheros que Git ignora debido a las reglas definidas en el archivo .gitignore.

Comandos bsicos para gestionar ficheros en Git

  1. Aadir un fichero al staging area:

    git add fichero

  2. Confirmar los cambios del staging area al repositorio:

    git commit -m "mensaje"

  3. Incluir en el commit los cambios en los ficheros staged y tambin en los modified:

    git commit -a -m "mensaje"

  4. Quitar un fichero del staging area (opuesto a git add):

    git restore --staged fichero
# Alternativa menos usada:
git reset fichero

  5. Quitar un fichero del staging area pero mantenerlo en el rea de trabajo:

    git rm --cached fichero

  6. Borrar un fichero del staging area y del rea de trabajo:

    git rm fichero

  7. Restaurar un fichero modificado en el rea de trabajo al estado del ltimo commit:

    git restore fichero
# Alternativa menos usada:
git checkout fichero

Comprobacin del estado de los ficheros:

Para ver el estado de los ficheros en las diferentes reas (untracked, staged, modified):

git status

El comando muestra:

  • Archivos en la staging area (cambios a ser confirmados).
  • Archivos modificados pero no aadidos al staging area.
  • Archivos sin seguimiento (untracked).

Ejemplo de salida:

En la rama main
Cambios a ser confirmados:
  (usa "git restore --staged <archivo>..." para sacarlos del rea de stage)
	nuevos archivos: facturas.html

Cambios no rastreados para el commit:
  (usa "git add <archivo>..." para actualizar lo que ser confirmado)
  (usa "git restore <archivo>..." para descartar los cambios en el directorio de trabajo)
	modificados:     main.js

Archivos sin seguimiento:
  (usa "git add <archivo>..." para incluirlos en el rea de stage)
	cliente.html

Notas importantes:

  • Si modificas un fichero despus de usar git add y no vuelves a aadirlo con git add, el commit solo incluir los cambios que estaban en el staging area cuando ejecutaste git add.
  • Para evitar este problema, se puede usar el comando git commit -a, que automticamente aade los ficheros modificados al staging area antes de confirmar los cambios.

Sincronizacin de Git y Ramas

Una misma rama en git puede estar en varios sitios a la vez. El repositorio es el histrico donde estn todos los commits que se han hecho.

Explicacin de las ramas suponiendo que estamos en master:

  • Carpeta local o rea de trabajo o rbol de trabajo: Es donde estn los ficheros con lo que trabajamos nosotros directamente mientras programamos.

  • Repositorio local: Es el repositorio que tenemos en nuestro ordenador.

    • Rama master: Es donde se guardan los ficheros al hacer commit
    • Rama origin/master: Es una copia que hay del repositorio remoto.

  • Repositorio remoto: Es un repositorio que est en otra mquina.

    • Rama master: Es donde queremos copiar los commits de nuestra rama master. Al hacer referencia a una rama de un repositorio remoto, se usara la expresin origin master con un espacio en medio.

Si queremos movernos entre ramas:

git switch master
git checkout master # Comando ms antiguo, se recomienda switch

Si queremos crear una nueva rama:

git branch <nombreRama>

Si queremos crear una rama y automticamente movernos a ella:

git switch -c <nombreRama>
git checkout -b <nombreRama> # Comando antiguo, se recomienda switch

Si queremos movernos a un commit en concreto con switch:

git switch --detach <hashDelCommit>

Si queremos movernos a una rama de tipo origin/rama debemos hacerlo con:

git switch --detach origin/rama
Note

DETACH nos informa de que no se puede crear commit al lugar al que nos movemos. Es decir, no podemos trabajar en ese lugar

Si queremos publicar los cambios de una rama:

git push origin master
git push

Si queremos obtener los cambios del remoto:

git fetch origin master
git fetch

# FETCH TIENE LA OPCIN --PRUNE PARA BORRAR LAS CARPETAS QUE HAN SIDO BORRADAS EN REMOTO
git fetch --prune

Si queremos obtener y aplicar en nuestra rea de trabajo los cambios del remoto:

git pull
Caution

git pull hace un fetch y un merge, lo que puede producir conflictos que posteriormente haya que solucionar y puede que el histrico se modifique.

Para mergear lo que hay en la rama origin/master en la rama master:

git merge origin/master

Si queremos eliminar una rama en local:

git branch -d <rama>

Si queremos eliminar una rama en remoto:

git push origin -d <rama>

Para fusionar ramas tenemos dos opciones:

git merge <nombreRama>
git rebase <nombreRama>

Diferencia entre Merge y Rebase?

Merge conserva la historia original de la rama fusionada y rebase reescribe la historia de la rama para que parezca que los commits se aplicaron de forma lineal. En otras palabras, merge fusiona e incorpora los cambios especificados a la rama actual (donde est el HEAD) mientras que rebase mueve el HEAD a la rama especificada.

Note

HEAD es donde estamos ubicados._

Remotes

En Git un "remote" hace referencia a un servidor de Git donde subimos el cdigo. Por defecto cuando hacemos git clone se crea un remote llamado origin.

Si queremos ver la lista de remotes:

git remote -v

Para aadir un remote:

git remote add otroservidor <URL>

Borrado de commits

GIT RESET modifica el estado actual de un fichero.

Tienes diferentes opciones:

  • --soft: Vuelve a un commit y conserva los cambios en el directorio de trabajo y en el stage.
git reset --soft <hashCommit>
  • --mixed: Los cambios se conservan en el stage.
git reset --mixed <hashCommit>
  • --hard: Los cambios no se conservan.
git reset --hard <hashCommit>
Borra los commits siguientes Se modifica rea de trabajo Modifica staged area
--hard Si Si Si
--mixed Si No Si
--soft Si No No
Note

La opcin HEAD~1 hace referencia al ltimo commit.

Deshacer commits

GIT REVERT deshace un commit y crea uno nuevo volviendo al estado del commit especificado.

git revert HEAD~1
git revert <hashCommit>
  • Prepara un nuevo commit que deshace el ltimo commit pero sin realmente hacer el commit. Lo que hace el modificar los ficheros y aadirlos a la staged area. Despus se hace el commit. La opcin --no-commit es til porque nos permite revisar que va a hacer el commit y nos permite hacer alguna modificacin ms. Si no estamos interesados en hacer el commit lo podemos abortar con git revert --abort o usar el ya conocido git restore --staged --worktree fichero.
git revert --no-commit HEAD~1
git commit -am "Mensaje de commit" # -a para aadir a stage & -m para el mensaje del commit
Note

git restore --staged --worktree fichero  restaura el archivo fichero tanto en el rea de preparacin como en el directorio de trabajo a su estado en el ltimo commit, eliminando cualquier cambio realizado.

Configuracin

.gitkeep

Por defecto Git no sube las carpetas vacas, por lo que si queremos que las suba hay que aadir algn fichero. Por convencin se suele crear un fichero llamado .gitkeep y al ya haber algn fichero, git subir esa carpeta.

.gitignore

El fichero .gitignore permite indicar que carpetas no se deben subir a git. El fichero debe estar en el raz del proyecto de git.

Almacenando contraseas

Normalmente trabajamos siempre en nuestro ordenador y no queremos todas las veces volver a poner la contrasea. En ese caso se puede decir a git que la almacene y no nos la vuelva a pedir:

  • No te vuelve a pedir la contrasea en 15 min
git config --global credential.helper cache
  • Almacena la contrasea en el disco en texto plano y nunca mas te la vuelve a pedir.
git config --global credential.helper store

Stash

Es un almacn que permite guardar los cambios de los que todava no se quiere hacer un commit.

  • Guardar solo lo modificado (estn o no en el staged area):
git stash push
  • Guarda todos los ficheros aunque NO estn en la stage area. Es decir, tambin los ficheros nuevos. Y al hacer el pop se restaura el estado en el staged area.
git stash push --include-untracked

Por defecto --include-untracked --all
Los ficheros modificados (estn o no en el staged area)
Los ficheros nuevos
Los ficheros los ignorados
  • Y si luego queremos recuperar los cambios se hace con:
git stash pop --index # Si no se aade --index, no se restaurar el staged area.

Log

El comando git log permite ver el histrico de commits.

  • Ver los commits en una nica lnea cada commit.
git --no-pager log --pretty=oneline
  • Ver los commits indicando que campos queremos mostrar e indicando el formato de la fecha.
git --no-pager log --pretty=tformat:"%h %cn %cd %s" --date=format:"%d/%m/%Y %H:%M:%S"

Mensajes de commit

Los mensajes de commit deben seguir el siguiente formato:

type(#issue):titulos

Explicacin (opcional)

Siendo

  • type
    • feat: Si se aade una nueva funcionalidad (feature)
    • fix: Si se arregla (fix) un error
    • docs: Si solo cambia cosas de la documentacin
    • style: Si solo se cambia el estilo del cdigo como tabuladores, puntos y comas, formateo, etc.
    • refactor: Si se cambia el cdigo para mejorar su calidad pero sin modificar la funcionalidad. Eso se llamada refactorizar.
    • test: Si se cambian cosas relacionadas con test automticos.
    • chore: Si se cambian cosas relacionadas con el despliegue.
  • #issue: Es el N de la incidencia a la que hace referencia.

Ejemplos:

  • Se arregla el fallo 45 que se llama "Falla si la fecha est vaca"
  fix(#45):Falla si la fecha est vaca
  • Se aade la nueva funcionalidad llamada "Mostrar el listado de los pacientes" con N 456
feat(#456):Mostrar el listado de los pacientes
